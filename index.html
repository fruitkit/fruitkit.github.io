Crypto Guide ğŸ¤¤

Source Publication Date: March 14, 2019

Public Key Encryption

Use, in order of preference:

  1. X25519 (for which the key size never changes) then symmetric encryption.
  2. ECDH with secp256r1 (for which the key size never changes) then symmetric encryption.
  3. RSA with 2048-bit keys.

You're better off not using RSA if you can help it.

ECDH: 256-bit keys RSA: 2048-bit keys

Public Key Signatures

Use, in order of preference:

  1. Ed25519 (for which the key size never changes).
  2. ECDSA with secp256r1 (for which the key size never changes).
  3. RSA with 2048-bit keys.

Everything said about RSA encryption applies to RSA signatures.

ECDSA: 256-bit keys RSA: 2048-bit keys

Symmetric Key Encryption

Use, in order of preference:

  1. XChaCha20-Poly1305 or XSalsa20-Poly1305 (which always have 256-bit keys)
  2. AES-GCM-SIV (regardless of key size)
  3. ChaCha20-Poly1305 (which always has 256-bit keys)
  4. AES-GCM (regardless of key size)

For application-layer symmetric-key encryption, two additional options should be considered.

  1. AES-CTR (regardless of key size) + HMAC-SHA2 (Encrypt then MAC)
  2. AES-CBC (regardless of key size) + HMAC-SHA2 (Encrypt then MAC)

Symmetric Key Authentication

Use HMAC with a SHA2-family hash function, with a key size equal to the hash function size.

Make sure you're using at least 224-bit keys for SHA-224.

Do NOT design your own message authentication protocol out of a hash function. Use HMAC.

If your symmetric encryption includes Poly1305 authentication, that's great, but it requires expert care to use it safely. Don't use Poly1305 standalone unless you're an expert.

224-bit, 256-bit, 384-bit, 512-bit are all good key sizes, provided your algorithm is reasonable.
